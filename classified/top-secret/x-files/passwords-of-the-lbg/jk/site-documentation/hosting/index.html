How we host our site.

TLDR: using GitHub Pages and Namecheap.

GitHub Pages

GitHub Pages is a service provided by GitHub that hosts any static content in a repository as a website. This is how we can just create a new .html file and have it turn into a page on the website. More on static sites on our structure page.

You can find the GitHub Pages settings under Settings > Pages. The settings should be as follows:

  1. Source: Deploy from a branch. This means the site will come directly from a branch, without any extra processing.
  2. Branch: master, Source: / (root). We deploy our site from the master branch and from the root folder.
  3. Custom domain: www.bestghent.eu. It is important to add the www prefix, otherwise GitHub gets confused. bestghent.eu gets redirected to www.bestghent.eu via Namecheap.
  4. Enforce HTTPS. This will make sure all traffic to our site is encrypted and safe.
Last but not least: In the repository itself, in the root folder, there needs the be a file called CNAME (without a file extension) containing the text www.bestghent.eu. This links to the CNAME Record in the list of host records at Namecheap.

Namecheap

As the name implies, Namecheap provides domain names for economic prices. Our domain, bestghent.eu, is registered using their cheapest, simplest plan. We don't pay for hosting, as we're hosted via GitHub Pages and we don't pay for an email service.

All we ask Namecheap to do is redirect all requests to bestghent.eu to our GitHub Pages setup. This is done by having the following host records under the Advanced DNS section on the dashboard:

  1. A Record @ 185.199.108.153
  2. A Record @ 185.199.109.153
  3. A Record @ 185.199.110.153
  4. A Record @ 185.199.111.153
  5. CNAME Record www best-ghent.github.io.
The A Records point our domain name to GitHub's servers. Those will redirect bestghent.eu to www.bestghent.eu and act as the servers serving the sites content. The CNAME Record points all www requests to our organisation's account on GitHub. This allows GitHub to know it is our side they need to serve to the client and not someone else's.

Help! The site is down!

In case the site is down, as long as GitHub itself is still up, this documentation can still be accessed by everyone. Simply clone the repository, and access the docs using the Live Server extension in VS Code.

Now, how to debug: There is of course no complete tutorial on how to debug something like this, but there are places you can start to look.

  1. A single page is down: This likely means there is only a problem with the file containing that page. Make sure the file is named index.html and is located in the correct folder.
  2. A page gives a completely blank screen. This sounds like a problem with the FOUC-barrier. You can check for error messages by using Chrome DevTools (can be opened by pressing ctrl+shift+c). Make sure you included the nescessary boilerplate correctly. A guide on this can be found here.
  3. GitHub: as we are hosted using their services, it makes sense to check if everything on their side is set up correctly. Maybe there is a general outage, maybe some settings were changed,... Make sure to check any emails from GitHub regarding changes to GitHub Pages.
  4. Namecheap: same deal with them. Make sure everything is set up correctly as described above, check for emails, ... If you're lucky, we just forgot to pay them and we can get the site back by renewing our subscription.
