How to: Git

Downloading/Installing

  1. First, download the (latest) Git installer from the official download page.
  2. Next, run the installer. The default options are usually fine.
  3. To check that Git is installed correctly, type git in the command prompt. If this gives you a list of git commands, you're set and VS Code should be able to detect and use your Git version.

A note on authentication

Git does not use "accounts" or "users". The only time any credentials are involved is when you try to push your changes to an external server, like GitHub. This means that when you push to GitHub for the first time, you will be asked to authenticate.

The confusing thing is how you're asked to authenticate. You will be presented with a prompt for username and password, and surprise: Filling in your GitHub username and password will not work. This is because what GitHub actually wants is your username and a token. These tokens usually have expiry dates as well. All in the name of security.

You can create a token for your account on GitHub's website. If you don't want to have to log in again every few months you might want to set the expiry date to never, even though GitHub does not recommend this.

Basic Git operations

There are a lot of things you can do using Git. While VS Code provides a nice interface to perform these operations, Git also has a CLI or Command Line Interface. A third party Git client like fork will of course also expose all these options.

Cloning


git clone [url_to_repository] [directory]
git clone https://github.com/BEST-Ghent/BEST-Website.git BEST-Website-folder

The clone command allows you to clone a remote repository (like the one containing our website) onto your computer. This includes the entire commit history and all branches.
You can also specify a directory to clone the files into. If you have already made a folder for this, you can leave this field empty and the repository will be cloned into your current working directory.

Diff and Status


git diff
git status

git diff shows the changes between your code/files and the last active commit that are not staged.
git status displays changes that are staged.
An example: When you have freshly cloned the website's repository, it's last commit will be the one used to display the public site. By default, this is the active commit. If you make changes to the files in your local folder, they will be displayed when using git diff. You can prepare these changes to be committed by staging them (more on staging next). When changes are staged, they will show up when when using git status. When you later commit these changes, they will disapear from the status, as they won't differ anymore from the (new) active commit.

Staging


git add [file]
git add training/32-jan-2025/index.html
git reset [file]
git reset training/32-jan-2025/index.html

To stage a newly created file or changes to a file, use git add. To unstage them, use git reset.

Now why stage anything? Well, staged changes can be solidified by committing them.

Commits


git commit -m “[descriptive message]”
git commit -m “Added a page for the next training.”

When you commit your changes, you create a snapshot of your code/files at a certain point in time. This snapshot is not a direct copy of your entire project, but rather a list of all the changes since the previous commit. In the example code above, there is also the option to give the commit a name/title. Many Git clients allow you to specify a title and a description. Whilst this can also be done using the CLI, the command becomes longer and more complicated.

It is important to note that git commit will only commit staged changes. If you forget to stage your changes, you will create an empty commit, so make sure to check that all your changes are staged by using git status.

Please give every commit you make a descriptive title. "Update index.html" for example, is not descriptive. "Update Deloitte description" is better and "Updated Deloitte description" is great :-) Good descriptions make it easier to track There is now still one final step before your changes reach the website: Pushing them to GitHub.

Pushing


git push [alias] [branch]
git push origin master

Changes, commits, branches,... are all part of a repository. But the repository on our computer is cloned (a copy) of the repository on GitHub. This means that when we commit something on our computer, we do it to our local repository. To push these changes to the repository on GitHub, we can use git push. A repository can link to other remote repositories via aliases. When a repository is cloned, a default alias called origin is created, linking to the original repository. When a branch is specified, only the changes to that branch will be pushed to the remote. It is common practice to always specify the branch to use when pushing, because when none is specified, it will try to push all branches, which is usually not intended. The master branch is the main branch where most of the development takes place. If you're new to all of this, creating new branches will probably not be nescessary for you.

Important: git push will ask for some authentication (as mentioned above). But on top of that, it will only work if your GitHub account has the nescessary permissions on GitHub. This is to prevent random people from pushing changes to the site. So make sure to ask someone to add your account to the repository on GitHub.

A note about the name master: At some point, the default branch name for new repositories on GitHub was changed from "master" to "main" because master was too kinky. The repository of our page was made before that change, hence the name of our master branch. Branches can be renamed, but there is not really a reason to do so. If someday, someone does take offense to the name master, they can change the name to whatever they think is better. My personal favourite main branch name is "home" by the way.

How to get the code of the website

Using the commands mentioned above, you could in theory figure out how to clone the site's repository to your computer. But if you just want to get started, you can follow this tutorial:

  1. Open the command prompt. You can search for cmd on Windows and for Terminal on mac and Linux.
  2. Set your working directory to the folder where you want to clone the code into. You can check your current working directory by typing cd on Windows or pwd on Unix based systems. You can google how to change your current working directory. Usually opening the command prompt will copy the directory you're currently in.
  3. To clone the sites code, use the following command: git clone https://github.com/BEST-Ghent/BEST-Website.git.
  4. Once the repository is cloned, you can open the folder in VS Code and start making changes. You can commit and push them using VS Code's built-in Git support.
