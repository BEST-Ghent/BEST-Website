How we structure our site.

Because we are hosted using GitHub Pages, we can only serve static content on our website. This effects how we structure the pages on our site and hence how we structure the files and folders in the codebase.

Static sites

As we are hosted using GitHub pages, we can only serve static content on our site. This means there is a direct correspondence between how files and folders are structured and the links to the pages on the website.

If we create a file index.html in a folder hello, we can find the corresponding page using the link bestghent.eu/hello/. We could also create two pages in the same folder, like hello1.html and hello2.html, but then we get the less aesthetically pleasing bestghent.eu/hello/hello1.html and bestghent.eu/hello/hello2.html. This makes index.html special, as it is the default page to be loaded when no file is specified in the url.

On this website, we only use index.html files to keep our links nice and tidy. This means we create seperate folders for every page we create. The only exception to this is the 404.html page in the root folder that is displayed when someone tries to access a page that doesn't exist.

The folders of pages with related content can share a parent folder. The tools/git/ and tools/vscode/ pages in this documentation are an example of this.

The static folder

Websites require a lot of images to display properly. Most of them also need JavaScript and Stylesheets and maybe even some pdf or mp3 files. All these files have to be stored in the repository to be accessible to the client, but where?

One approach for storing these assets could be the following: If you have a page like training/index.html, you could store your assets in that folder: training/img1.png. This works well for images that are only used once on the entire site. Now imagine we have a profile picture of a board member boardie.png: This image could be used in the about page, in a members page, maybe on the page of an event they are hosting, ... This approach would mean copying the same image multiple times. It also means you can't easily categorize images by type (like pfp's) and easily find the one you're looking for.

This is why we went with a centralized approach: The static folder.

Every file that is not a page (i.e. an index.html file) is stored in the static/ folder. There are only 3 exceptions to this rule: The CNAME file, the 404.html page and the README.md. The CNAME file links the repository to our domainname (bestghent.eu) and has to be in the root folder to serve its purpose. The README file serves as a description of the repository to newcomers and will only be displayed by GitHub when in the root folder.

Technically there is another exception to this rule. This documentation has its own static folder containing all files that are not related to the main site. The documentation does use the main static folder to access the elements it has in common with the main site like stylesheets and scripts. This second static folder exists to seperate documentation files from the main site and because the documentation is considered it's own website and therefore redefines the root folder (more information here).

Subdivision of the static folder

Just dumping every file that is not a page into the static folder is sadly not good for most people's sanity. Because of this, the static folder is subdivided into 3 more folders: scripts, stylesheets and media.

To this day, all files on this site could always be categorized into one these three folders. If this ever stops being the case, you could create a new category for these new files. Or, if this category-defying menace of a file is too unique to be categorized, drop it straight into static/ itself.

static/scripts/ and static/stylesheets/

All JavaScript files should go in the scripts folder and all CSS files should go in the stylesheets folder. This is to distinguish vital code that affects the entire page from more individualized assets that only affect their general viscinity. More on JavaScript and Stylesheets here.

At the point of writing, the scripts and Stylesheets in these folders are so generic, general and scarce in numbers that they are not sorted into further subfolders. If more subdivisions ever become nescessary, feel free to make them, but be wary of the final note at the bottom of this page on moving files.

static/media/

As the name implies, all images, pdf's, sound effects, ... go here. The media folder itself also has some categories to further structure the files inside. The most important ones are static/media/pages/ and static/media/people/.

  1. static/media/pages/

    This folder is for the media that is specific to a single page. This means that files that are unique to the about page will be in static/media/pages/about. An example is the banner on the homepage. This is a picture of someone wearing a BEST sweater at a lake. It is only used by the homepage, on the homepage and therefore resides in static/media/pages/home/.

    It can be tempting when creating a new page to put every new image into that new page's media folder. The problem is that if this file is later deemed cool enough to be used elsewhere (which is very common with images of people/groups f.e.), it is still linked to that one page. So in that case the file must be moved to a different location, which poses the risk of breaking pages that are currently using it (see the note on moving files below).

    So where does a file go if it is not unique to a single page? For these files there are other folders in static/media/, like the people folder. Just as with the main static/ folder, files that are too unique to be categorized can just be dumped into static/media/ itself.

  2. static/media/people/

    Every image or document that is related to a person goes into the folder of that person in here. For example: A profile picture for the autumn course of 2025 of Emmanuel Obiringa will go into static/media/people/Emmanuel Obiringa/Emmanuel_pfp_autumn_course_2025.jpg. Know that there will be many images with similar meanings in one person's folder, so be specific when naming them. Just Emmanuel_pfp.jpg is not specific enough, as he might want a different pfp for different events. Renaming files can cause a mess (see the note on moving files below), so try to get the name right first try.

Unused pages

A page is considered unused when: The page is not (meant to) be accessed by anyone anymore, won't be missed in any way if removed and, importantly, there are no links anywhere on the website that link to it anymore. If all of these criteria are met, you can remove that page (delete the index.html and the folder that contained it) from the site.

Removing unused pages helps keeping the codebase contained in size and should be done whenever a page becomes unused. It is important to know that removing a page from the code base does not remove it from the commit history. You will always be able to access anything that has ever been part of the website by checking out a commit from that time using Git. So don't worry when removing something: It was archived the moment it was added to the site. This is also why it is important to never add passwords or other sensitive information to the website, because it will be technically be accessible until the site's repository itself is removed from GitHub.

If a page is unused or not can sometimes be a bit of a gray area. Think of the page of an event from a long time ago. Is it still used? No, but does it hurt to still exist? Also no. In cases like this, it might be wise to discuss the removal of that page with the President, or at an LV.

Pages of local courses should remain accessible to the participants by their original link. They should not be relocated (as that would break the link) and are never considered unused. This is why all local course pages go into the localcourses/ folder. This prevents them from cluttering the root directory.

Broken pages

When making changes to the website, it is good practice to check if those changes unintentionally affected other pages than the one you worked on. This can be especially tricky when tinkering with the template system, as almost all pages rely on it. Still, it can happen that a page breaks because someone forgot to check if their changes broke something or simply because programming is hard and the internet is ever-changing.

A page is considered broken when content doesn't render (correctly), widgets like links, buttons, ... don't work anymore, dynamic content like the events in the calendar is not retrieved correctly or anything else is not as it is supposed to be.

If a page is broken, it should either be fixed or removed. For active pages it is clear that they should be fixed, and unused pages should already be removed due to being unused. But for pages where it's not clear if they are unused or not, their level of disrepair can be a deciding factor: If it is not worth the effort to repair a page that is already not really used anymore, don't repair it.

Rule summary and good practice

Please follow these rules as described above. They exist to keep the site structured and easy to navigate for both the client and yourself.

  1. All page files are called index.html.
  2. All non-page files, aside the exceptions mentioned above, reside in the static/ folder.
  3. All JavaScript files go into static/scripts/. All Stylesheets go into static/stylesheets/.
  4. All images, pdf's, sound effects, ... go into static/media/
  5. Use the static/media/pages/ directory as little as possible by categorizing files correctly.
  6. Every file that is related to a person goes into that person's static/media/people/ folder.
  7. Remove unused pages.
  8. Always repair or remove broken pages.
If you feel like there is a rule missing, or want to change an existing rule, add them to this page with an explanation about why the rule exists. Before changing existing rules, make sure you don't hinder the original intentions of that rule (like the static folder preventing file repetition).

Also make sure the new rule is applied consistently to the entire codebase and not only to new files. f.e. when you make a rule saying all folder names should start with the letter 'a', make sure to also rename all existing folders accordingly. If only half the codebase uses a rule, then it's not a rule but a mess.

A note on moving files

The largest disadvantage of sharing a file among many different pages is that when you delete, move, rename,... that file, all of the pages that reference it will not be able to find it anymore. This means you should really think about the name and place of new files when you add them, because moving them later might be a hassle and could break many different pages at once.

You can still do this if you think a restructuring of the files is nescessary, but make sure that, afterwards, you check every single page for missing files/being broken.
